<objective>
Extend `@ags/shared` types to support 1→1 article equivalencias and the `'conversion'` subtipo on `MovimientoStock`. This plan is **strictly additive** — every new field is optional. No existing consumer breaks.

Purpose: Unlock STKE-02..07. All downstream service + UI plans import from `@ags/shared`; landing the types first means downstream plans cannot land before this.

**Plan summary — minimum MovimientoStock surface (revised per checker M6):**

CONTEXT.md (line 64) locks **one** new optional field on `MovimientoStock`: `subtipo?: 'conversion'`. To support `desagregarUnidades` audit, this plan adds **three additional companion fields** beyond the locked literal — kept to the minimum needed by 13-03's runTransaction:

- `subtipo?: 'conversion'` (CONTEXT-locked literal)
- `articuloDestinoId?: string | null` (queryable destino-side history per research Open Question #3)
- `cantidadDestino?: number | null` (numeric count of the destination side — cannot be reconstructed without it)
- `factorConversion?: number | null` (record the factor at operation time so historical movs are stable when factor is later changed)

**Dropped from earlier draft** (per M6 trim — keep them out of the type):
- `articuloDestinoCodigo` — denormalize at audit-display time from `articuloDestinoId` instead of persisting
- `articuloDestinoDescripcion` — collapse into the human-readable `motivo` string instead of a separate field

If the user requests the denormalized fields back for audit-display performance, re-introduce them in a follow-up phase with explicit endorsement.

Output:
- New exported interface `ArticuloEquivalencia`
- Updated `Articulo` interface with `equivalencias?: ArticuloEquivalencia[]` AND `articuloIdDestinoEquivalencia?: string | null` (flat field for query indexing)
- Updated `MovimientoStock` with `subtipo?: 'conversion'` + `articuloDestinoId?: string | null` + `cantidadDestino?: number | null` + `factorConversion?: number | null` (4 new optional fields total, all back-compat)

```ts
/**
 * Equivalencia 1→1 entre código de compra y código de uso (Phase 13).
 *
 * Vive en el artículo de COMPRA (el origen) y apuntando al artículo de USO (el destino).
 * Sentido único; el reverso (uso→compra) se deriva en runtime como `1 / factor`.
 *
 * Campos denormalizados (`articuloCodigoDestino`, `articuloDescripcionDestino`) se persisten
 * para evitar joins en listas; se refrescan via `articulosService.update()` cuando el destino se renombra.
 */
export interface ArticuloEquivalencia {
  /** FK → articulos (del lado de uso) */
  articuloIdDestino: string;
  /** Desnormalizado para display sin join */
  articuloCodigoDestino: string;
  /** Desnormalizado para display sin join */
  articuloDescripcionDestino: string;
  /** Factor multiplicativo: cantidad destino = cantidad origen × factor. Acepta decimales (ej. 0.1, 0.5, 10, 100). */
  factor: number;
}
```

3. **Inside** the existing `Articulo` interface, just before the closing `}`, add two new optional fields (after `resumenStock`):

```ts
  /**
   * Phase 13 STKE-01 — vinculación 1→1 con artículo de uso.

   * Use `articulosService.linkEquivalencia/unlinkEquivalencia` para mutarlo — no escribir directo.
   */

  /**
   * Phase 13 STKE-01 — campo plano denormalizado de `equivalencias[0]?.articuloIdDestino`.
   * Existe sólo para que Firestore pueda indexar `where('articuloIdDestinoEquivalencia', '==', X)`
   * — Firestore NO soporta `array-contains` sobre propiedades dentro de un objeto del array.
   * Source-of-truth es `equivalencias[]`; este campo es siempre derivado y se mantiene en sync.
   * `null` cuando no hay equivalencia configurada.
   */
  articuloIdDestinoEquivalencia?: string | null;
```
