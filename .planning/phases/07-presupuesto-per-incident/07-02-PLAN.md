<objective>
Implementar token-first order en envío por mail: validar/obtener OAuth token ANTES de cambiar estado en Firestore. Si el token falla o `sendGmail` falla, el presupuesto NO transiciona a `enviado`. Al éxito, hacer un único `updateDoc` atómico con `estado: 'enviado' + fechaEnvio: today`.

Purpose: Cierra Pitfall 5-A documentado en STATE.md. Hoy el flujo es: el usuario clickea "Enviar email" → `EnviarPresupuestoModal.handleSend()` corre `requestToken → generatePDF → sendGmail → onSent()` → y el callback `onSent` en `EditPresupuestoModal` (líneas 322-330) es quien llama `handleEstadoChange('enviado')` + `save()`. Si el token falla, `handleSend` throwea antes de llamar `onSent`, OK — pero hay otros bugs: (a) el `save()` dentro de `onSent` recorre TODO el form y re-serializa ítems/cuotas/etc, no es un update quirúrgico — si algo del form está en estado inconsistente, puede fallar post-envío y dejar el doc en `borrador` con el mail ya despachado; (b) el cambio de estado está desacoplado del resultado real del envío — si por alguna razón `onSent` no se invoca (ej. el component unmountea antes), el estado nunca cambia; (c) no hay manejo diferenciado de errores por etapa, todos muestran el mismo `error` genérico.

Output: Flow de envío atómico y confiable, reutilizable para `'partes'`, `'mixto'`, `'ventas'` de Phase 10.
</objective>

**FINDING-A (CRITICAL — FMT-02 core):** Hoy el cambio de estado vive en `EditPresupuestoModal.onSent` (líneas 322-330), separado del `EnviarPresupuestoModal.handleSend`. Esto genera 3 problemas:
  - (A1) Si `handleSend` arroja (token fail o mail fail), nunca se llama `onSent` → OK, estado no cambia. Este sub-caso ESTÁ OK por accidente.
  - (A2) Si `handleSend` completa OK PERO el `save()` dentro de `onSent` falla (Firestore down, validación del form corrupta, etc.) → el mail ya salió y el estado queda en `borrador`. Inconsistencia silenciosa.
  - (A3) El `save()` en `onSent` re-serializa TODO el form (items + cuotas + adjuntos + condiciones + responsable + ...), no hace un update quirúrgico de `{estado, fechaEnvio}`. Si el form tiene un estado inconsistente (ej. un ítem con categoria inválida), `save()` puede corromper el doc tras un envío exitoso.

**FINDING-B:** `EnviarPresupuestoModal` no diferencia errores por etapa. El `try/catch` en líneas 68-111 captura todo al mismo `setError(...)`. Si el usuario cancela el popup OAuth, ve "Error desconocido"; si la red falla en `sendGmail`, ve el mismo mensaje genérico. UX: el mensaje + retry deben ser específicos por etapa.

**FINDING-C:** El orden actual de `handleSend` es:
  1. `requestToken()` (línea 74)
  2. `generatePresupuestoPDF()` (línea 79)
  3. `sendGmail()` (línea 87)

**Step 2 — Reescribir `handleSend` con order estricto y manejo de errores por etapa:**

```tsx
import { presupuestosService } from '../../services/firebaseService';
// ... existing imports

const handleSend = async () => {
  if (!to.trim()) { alert('Ingrese al menos un destinatario'); return; }

  setSending(true);
  setError('');

  // ── STAGE 1: OAuth token ──
  let accessToken: string;
  try {
    setStatus('authorizing');
    accessToken = await requestToken();
  } catch (err) {
    const msg = err instanceof Error ? err.message : 'Error desconocido autorizando Gmail';
    setError(`No se pudo autorizar Gmail: ${msg}. Podés reintentar.`);
    setStatus('error');
    setSending(false);
    return;  // NO Firestore write. Estado NO cambia.
  }

  // ── STAGE 2: Generate PDF ──
  let pdfBase64: string;
  try {
    setStatus('generating_pdf');
    const { generatePresupuestoPDF } = await import('./pdf');
    const pdfBlob = await generatePresupuestoPDF(pdfParams);
    pdfBase64 = await blobToBase64(pdfBlob);
  } catch (err) {
    const msg = err instanceof Error ? err.message : 'Error generando PDF';
    setError(`No se pudo generar el PDF: ${msg}. Podés reintentar.`);
    setStatus('error');
    setSending(false);
    return;  // NO Firestore write.
  }

  // ── STAGE 3: Send Gmail ──
  try {
    setStatus('sending');
    const toList = to.split(',').map(e => e.trim()).filter(Boolean);
    const ccList = cc ? cc.split(',').map(e => e.trim()).filter(Boolean) : [];
    await sendGmail({
      accessToken,
      to: toList,
      cc: ccList.length > 0 ? ccList : undefined,
      subject,
      htmlBody: body,
      attachments: [{
        filename: `${presupuestoNumero}.pdf`,
        mimeType: 'application/pdf',
        base64Data: pdfBase64,
      }],
    });
  } catch (err) {
    const msg = err instanceof Error ? err.message : 'Error enviando mail';
    setError(`El mail no se pudo enviar: ${msg}. El estado NO cambió — podés reintentar.`);
    setStatus('error');
    setSending(false);
    return;  // NO Firestore write. Mail NOT sent.
  }

  // ── STAGE 4: Mark as enviado (atomic) — only if currently 'borrador' ──
  if (presupuestoEstado === 'borrador') {
    try {
      await presupuestosService.markEnviado(presupuestoId);
    } catch (err) {
      const msg = err instanceof Error ? err.message : 'Error actualizando estado';
      // Mail WAS sent successfully — distinct UX message.
      setError(`Mail enviado OK, pero hubo un error actualizando el estado del presupuesto: ${msg}. El presupuesto sigue en "borrador" — cambialo manualmente.`);
      setStatus('error');
      setSending(false);
      // Do NOT return — let user see the error but also know the mail succeeded.
      return;
    }
  }

  // ── STAGE 5: Success ──
  setStatus('sent');
  setTimeout(() => {
    onSent();
    onClose();
  }, 1500);
  setSending(false);
};
```
