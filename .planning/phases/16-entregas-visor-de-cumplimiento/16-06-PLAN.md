pnpm dev:modular:electron

**UI-01 — Edición inline OT# guarda en presupuestoItem.otNumeroVinculada:**
1. Crear un presupuesto borrador con 1+ items (cualquier tipo).
2. Aceptarlo (botón Aceptar en EditPresupuestoModal).
3. Navegar a `/entregas` (Sidebar → Stock → Compras → Entregas).
4. Localizar la fila del item recién creado.
5. Click en celda OT# → escribir "25661" → presionar Tab (blur).
6. Esperar 1-2s.
7. Refrescar F5.
8. **Esperado**: el OT# "25661" sigue visible. Verificar también en Firestore (DevTools): el item del presupuesto tiene `otNumeroVinculada: "25661"`.

---

**UI-02 — Filtros persisten en URL al navegar:**
1. En `/entregas`, aplicar 2 filtros: Cliente="Bayer" + Semáforo="Rojo".
2. Verificar que la URL contiene `?clienteId=...&semaforo=rojo` (o equivalente).
3. Click en el link del presupuesto (columna Presupuesto#) → navega a `/presupuestos/<id>`.
4. Browser Back.
5. **Esperado**: los filtros están intactos (mismo state que antes del click). La URL todavía contiene los query params.

---

**UI-03 — Badge "Sin ETA" para presupuestos legacy:**
1. Buscar en `/entregas` un presupuesto aceptado **antes** del cutover (fechaAceptacion = null).
2. **Esperado**: las filas de ese presupuesto muestran columna ETA = "—" y columna Días = "Sin ETA" (gris), badge sin color crítico. La fila se renderiza sin warnings de React (DevTools console limpio).

Si no hay presupuestos legacy en dev, simular: en Firestore, abrir un presupuesto aceptado y borrar manualmente el campo `fechaAceptacion`. Reload → fila debe mostrar "Sin ETA" gris.

---

**UI-04 — Bulk-apply "Aplicar a todos los items":**
1. Crear un presupuesto borrador con 3+ items.
2. Abrir EditPresupuestoModal.
3. Click en botón "Aplicar a todos los items (N)" arriba de la tabla.
4. En el modal: seleccionar `A importar` + ETA = 45 → click Aplicar.
5. **Esperado**: los 3+ items reflejan `disponibilidad='a_importar'` y `etaDiasEstimados=45` en la tabla. Guardar el presupuesto. Reload. Valores persistidos.

**How to verify (pre-flight + command):**

**Pre-flight (el usuario debe confirmar)**:
- [ ] Working tree limpio (`git status` sin cambios non-stashed).
- [ ] En branch `main`, último commit es el merge de Phase 16.
- [ ] `apps/sistema-modular/RELEASE-CHECKLIST.md` revisado (5 min): login, módulos críticos, PDFs, Excel.

**Comando del release** (copy-paste exact):

```bash
# 1) Bumpear versión + crear tag local
pnpm --filter @ags/sistema-modular release:minor

# 2) Push del bump + del tag (asumir que release.cjs ya creó sistema-modular-v<x.y.z>)
git push origin main
git push origin sistema-modular-v<x.y.z>     # reemplazar <x.y.z> con la versión del package.json
```

**Bump rationale**: MINOR porque Phase 16 agrega features user-visible (nuevos campos en editor + nueva vista /entregas) sin breaking changes (todos los campos nuevos son opcionales; legacy compatible).

**Post-push**: La GH Action `release-sistema-modular.yml` se dispara con el tag, builda en `windows-latest`, publica el release. Las PCs instaladas reciben popup "Reiniciar ahora" en los próximos minutos.

**NO ejecutar acá — el usuario corre el comando localmente** porque:
1. Tiene el `.exe` en su máquina para smoke real antes del push.
2. El push del tag dispara broadcast a PCs instaladas → requiere gate humano explícito.
3. Precedente en `.claude/rules/release-flow.md` + Phases 14-08, 15-03.

Si CI falla post-push (en GH Actions), revisar:
- `.env.local` secrets en GH (VITE_FIREBASE_*, VITE_GOOGLE_*).
- CSP en electron/main.cjs si se agregaron dominios externos.
- Tag apuntando al commit correcto (`git log --oneline -1 sistema-modular-v<x.y.z>`).
